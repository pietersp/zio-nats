# Documentation Guide

This file defines how documentation for zio-nats is written and structured.
It is the source of truth for anyone adding or editing pages on the docs site.

---

## Philosophy

The reader's time is the scarcest resource. Give them what they need *now*, not everything at once.

- **Show before tell.** A working example comes before any prose explanation.
- **Progressive disclosure.** Start simple. Link to deeper pages. Never front-load complexity.
- **Each page has one job.** A page that covers two concepts should be two pages.
- **Assume ZIO knowledge, not NATS knowledge.** The reader knows `ZLayer`, `ZIO`, `ZStream`,
  `for`-comprehensions, and typed errors. They may not know what JetStream, KV, or consumers are.
  Define NATS concepts the first time they appear on any page. Never define ZIO concepts.

---

## Writing Rules

These are checkable. Every page should pass all of them.

1. **Lead with a working example.** The first substantive content on any guide page is a code
   snippet that compiles (via mdoc) and does something real. The snippet comes *before* prose
   explanation.

2. **Walk through the example.** After every non-trivial code block (more than ~5 lines), add a
   numbered walkthrough explaining each key line in plain English. Example:

   ```
   1. `nats.subscribe[String](Subject("greetings"))` — opens a typed ZStream; each payload is
      decoded from UTF-8 via the built-in `NatsCodec[String]`.
   2. `.take(3)` — the stream completes after 3 messages; the scope closes automatically.
   3. `.fork` — runs the subscriber in the background so we can publish below.
   ```

3. **One concept per section.** If a section starts covering a second concept, split it.
   Section titles should be skimmable — the title alone tells you whether to read it.

4. **Link, don't repeat.** If concept X is explained on page Y, link to Y instead of
   re-explaining. The only exception is a one-line reminder with a link:
   _"See [Serialization](./guides/02-serialization) for typed payloads."_

5. **Avoid jargon chains.** Don't write "a durable push consumer bound to a filtered subject".
   Write "a named consumer that remembers its position — like a named cursor in a stream."
   Link to the reference page for precise field names.

6. **Short paragraphs, short sentences.** Max 3 sentences per paragraph. If you need a 4th,
   start a new paragraph. No sentence should need to be re-read.

7. **Concrete over abstract.** Use real type names and field names. Write `env.value`, not
   "the decoded payload wrapped in the envelope".

8. **Code must compile.** All Scala code blocks in `docs/` are processed by mdoc. Write them
   to compile, or use `mdoc:silent` / `mdoc:fail` / `mdoc:reset` annotations intentionally.
   Never paste a snippet that you haven't verified compiles.

9. **Prefer `ZIO.service[X]` over constructor injection.** Examples should look like real ZIO
   apps — acquire services from the environment and wire with `provide` / `ZLayer`.

10. **Reference pages are different.** Config and API reference pages may be exhaustive tables
    or param-by-param lists. These do not need walkthroughs. They need accuracy and completeness.

11. **No hedging.** Don't write "you might want to", "it is possible to", "in some cases".
    Write "do X when Y".

12. **End guides with next steps.** The last section of every guide page is "Next steps" with
    2–4 links to related guides or reference pages.

## Site Structure

The Docusaurus sidebar is autogenerated from the `docs/` directory (see `website/sidebars.js`).
Numeric prefixes control ordering within a directory.

```
docs/
├── index.md                      ← Landing page
├── 01-quickstart.md              ← 5-minute tutorial
├── concepts/
│   └── 01-architecture.md        ← Service hierarchy + ZLayer graph
├── guides/
│   ├── 01-pubsub.md              ← Publish, subscribe, queue groups, request-reply
│   ├── 02-serialization.md       ← NatsCodec: built-in → zio-blocks → jsoniter → play-json
│   ├── 03-jetstream.md           ← Streams, publishing, consumers (durable + ordered)
│   ├── 04-key-value.md           ← KV operations, watch, TTL
│   ├── 05-object-store.md        ← Object store put/get, streaming, watch
│   ├── 06-configuration.md       ← NatsConfig, env vars, HOCON, auth, TLS
│   ├── 07-connection-events.md   ← lifecycleEvents, reconnection, status utilities
│   └── 08-testing.md             ← NatsTestLayers, testkit setup, integration test patterns
└── reference/
    ├── 01-nats-config.md         ← Full NatsConfig field table
    ├── 02-error-handling.md      ← NatsError ADT, catchAll patterns
    └── 03-modules.md             ← sbt artifact names + what each module provides
```

### Page intents

**`index.md` — Landing page**
3–5 sentences. One "Hello World" pub/sub snippet. One paragraph on what makes it different
(idiomatic ZIO, typed errors, zero raw jnats). Two calls to action: "Quickstart →" and
"Browse guides →". Nothing else.

**`01-quickstart.md` — Quick start**
End-to-end: zero to first message in 5 minutes. Sections: Prerequisites, Installation,
Connect, Publish & Subscribe. Must work copy-paste with a running NATS server (`nats-server`
or Docker). Walkthrough after every snippet.

**`concepts/01-architecture.md` — Architecture**
The `ZLayer` graph showing how all services derive from `Nats`. A brief description of what
each service is responsible for. No code required — this is a mental model page. Links to
the relevant guide from each service description.

**Guides**
Each guide follows the page template (below). Starts with a one-line summary, then covers
scenarios from basic → intermediate → edge cases. Ends with "Next steps" links.

**Reference pages**
Exhaustive. Tables preferred over prose. Every field, every error variant, every config option.

## Page Template

Use this as the starting point for every new guide page:

```markdown
---
id: <kebab-case-id>
title: <Title>
---

# <Title>

> One-sentence summary of what this page covers.

## Prerequisites

- [Quick start](../01-quickstart) completed
- Any extra requirements (e.g. JetStream requires `nats-server -js`)

## Basic example

\`\`\`scala mdoc:silent
// A minimal, complete, working snippet
\`\`\`

**What's happening:**

1. Explanation of line 1
2. Explanation of line 2

## <Next concept or scenario>

...

## Next steps

- [Link to related guide]
- [Link to reference page]
```

## mdoc tips

- `mdoc:silent` — compiles the block but does not show output in the rendered page.
  Use for setup code (imports, layer definitions) that would otherwise clutter the output.
- `mdoc:fail` — the block is expected to fail to compile. Use to demonstrate invalid usage.
- `mdoc:reset` — resets the mdoc session (all prior definitions dropped). Use at the start
  of unrelated examples on the same page to avoid name conflicts.
- `@VERSION@` — mdoc substitutes this with the current library version (from `mdocVariables`
  in `build.sbt`). Use in installation snippets:
  ```scala
  libraryDependencies += "io.github.pietersp" %% "zio-nats" % "@VERSION@"
  ```
- Run `sbt docs/mdoc` locally before pushing. The CI pipeline will fail if any snippet does
  not compile.
